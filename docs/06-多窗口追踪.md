# 06 - 多窗口/多会话追踪

> 用户可能同时开多个 Claude Code 窗口（不同终端、不同项目）。Touch Bar 只有一根，App 主窗口要展示全部会话。
>
> **核心问题：怎么决定"Touch Bar 当前显示哪个会话"？**

## 模型：单 Touch Bar + focused session

Touch Bar 是 Mac 全局资源（一根 bar），所以共享一个 HUD。但 HUD 里的 `cwd` 和 `status` 字段绑定到**一个 focused session**。

`model` / `cost` / `provider` 是全局的（同一份 `~/.claude/settings.json` 或 cc switch db），不分 session。

## 两个 Trigger 协同决定 focused session

```
Trigger 1: OS 窗口焦点变化（被动浏览）
  - cmd+Tab / 点窗口 → NSWorkspaceDidActivateApplicationNotification
  - 查映射: frontmost app + active tab → session_id
  - 命中 → focused session 切换

Trigger 2: Hook 活动（主动输入）
  - 用户在某个 CC 里输入 → UserPromptSubmit 触发
  - 更新 session.lastActivityAt = now
  - 该 session 自动成为 focused

优先级：Trigger 2 > Trigger 1
（用户在哪个 session 输入，就显示哪个 —— 信号更强）
```

### 为什么两个都要

| 场景 | Trigger 1（焦点） | Trigger 2（活动） | 单独够吗 |
|---|---|---|---|
| 用户切到 A 窗口但没输入 | 命中 | 不触发 | 1 够 |
| 用户在 B 窗口刚输入 | 不一定（B 可能已经在前面） | 命中 | 2 够 |
| 同一窗口多个 CC（tmux 多 pane） | 失效（窗口没变） | 命中 | 必须有 2 |
| VS Code 多个集成终端 | 失效（VS Code 不暴露 tab ID） | 命中 | 必须有 2 |
| 后台 streaming，用户在看别的 | 不命中 | 命中（streaming session 持续触发 MessageDisplay） | 2 显示"后台还在跑"，符合预期 |

**结论：** Trigger 2 是底座（必须），Trigger 1 是锦上添花（在用户没输入时也能跟随窗口）。

## Trigger 2 实现：活动时间戳

详见 [05-hook-接入层.md](05-hook-接入层.md)。每次 hook 事件到达：

```swift
state.sessions[event.sessionID]?.lastActivityAt = Date()
```

派生态：

```swift
var focusedByActivity: SessionID? {
    state.sessions
        .filter { $0.value.endedAt == nil }
        .max(by: { $0.value.lastActivityAt < $1.value.lastActivityAt })?
        .key
}
```

## Trigger 1 实现：OS 窗口焦点 + 终端 tab ID

### 1. NSWorkspace 监听

```swift
NSWorkspace.shared.notificationCenter.addObserver(
    self,
    selector: #selector(frontmostAppChanged),
    name: NSWorkspace.didActivateApplicationNotification,
    object: nil
)
```

### 2. 拿到 frontmost app 后查 active tab

| 宿主 | AppleScript / CLI | 返回值 |
|---|---|---|
| **iTerm.app** | `tell application "iTerm" to get id of session of current tab of current window` | `$TERM_SESSION_ID` 同款 ID |
| **Terminal.app** | `tell application "Terminal" to get name of selected tab of front window` | tab 名（需特殊处理） |
| **Ghostty** | `ghostty +list-sessions --format json` + 焦点匹配 | pane ID |
| **WezTerm** | `wezterm cli list --format json` + 焦点匹配 | pane ID |
| **tmux**（任何宿主） | `tmux display -p '#{pane_id}'` + `tmux list-panes -F '#{pane_id} #{pane_active} #{window_active}'` | `$TMUX_PANE` 同款 |
| **VS Code** | 无稳定 ID；用窗口标题匹配 `cwd` basename | 模糊 |
| **Cursor / Windsurf** | 同 VS Code | 模糊 |

### 3. 查映射表

```swift
// SessionStart 时建立
var sessionByTab: [TermTabKey: SessionID] = [:]

struct TermTabKey: Hashable {
    let hostApp: String       // "iTerm.app" / "Apple_Terminal" / "vscode" / "ghostty" / "WezTerm"
    let tabId: String         // $TERM_SESSION_ID / $TMUX_PANE / wezterm pane / VS Code 窗口标题
}
```

### 4. 命中后切 focused

```swift
func frontmostAppChanged() {
    guard let frontApp = NSWorkspace.shared.frontmostApplication else { return }

    let hostApp = mapBundleIdToHostApp(frontApp.bundleIdentifier)
    guard let tabId = queryActiveTabId(hostApp: hostApp) else {
        // 查不到 tab ID（如 VS Code），不切，让 Trigger 2 兜底
        return
    }

    let key = TermTabKey(hostApp: hostApp, tabId: tabId)
    if let sid = sessionByTab[key] {
        state.focusedByWindow = sid
    }
}
```

## 优先级合并

```swift
var focusedSessionID: SessionID? {
    // Trigger 2 优先：2 秒内有 hook 活动的 session
    let now = Date()
    if let recent = state.sessions
        .filter({ $0.value.endedAt == nil && now.timeIntervalSince($0.value.lastActivityAt) < 2 })
        .max(by: { $0.value.lastActivityAt < $1.value.lastActivityAt }) {
        return recent.key
    }

    // 否则用 Trigger 1
    return state.focusedByWindow ?? focusedByActivity
}
```

**2 秒窗口：** 用户切窗口后如果 2 秒内没输入，就认为他在浏览，跟随窗口焦点。一旦输入，立刻切到输入的 session。

## 跨终端可靠度矩阵

| 终端 | Trigger 1 可靠度 | 备注 |
|---|---|---|
| iTerm.app | ⭐⭐⭐⭐⭐ | AppleScript 拿 TERM_SESSION_ID，和 hook 里读的是同一个 |
| Terminal.app | ⭐⭐⭐⭐⭐ | 同上 |
| tmux（任意宿主） | ⭐⭐⭐⭐⭐ | tmux 自己的 pane ID 最稳 |
| Ghostty | ⭐⭐⭐⭐ | `ghostty +list-sessions` |
| WezTerm | ⭐⭐⭐⭐ | `wezterm cli list` |
| VS Code 集成终端 | ⭐⭐ | 无稳定 tab ID，用窗口标题模糊匹配 basename(cwd) |
| Cursor / Windsurf | ⭐⭐ | 同 VS Code，且可能更糟 |

**实用策略：** v2 先做 iTerm + Terminal.app + tmux（覆盖 80% 场景），其它终端 fallback 到 Trigger 2。

## "窗口有多个对话" 的具体场景

| 场景 | Trigger 表现 | 处理 |
|---|---|---|
| tmux 多 pane，每个跑 CC | Trigger 1 失效（窗口不变），Trigger 2 命中 | 用户在哪个 pane 输入，HUD 显示哪个 |
| VS Code 多个集成终端 | 同上 | 同上 |
| 同一 CC 用 `/resume` 切对话 | 每次 `/resume` 触发 SessionStart + 旧 SessionEnd | 映射表自动更新 |
| 用户关掉某窗口 | SessionEnd 触发 | 该 session 标记 ended，1 小时后 GC |
| 后台跑长任务，用户切到别的 CC | MessageDisplay 持续触发 | HUD 跟随后台 streaming session（符合预期） |

## SessionStart 时建立映射（关键）

详见 [05-hook-接入层.md](05-hook-接入层.md)。hook 推上来的 `_cc_touchbar` 块里有 `term_program` / `term_session_id` / `tmux_pane`：

```swift
case "SessionStart":
    let s = event.ccTouchbar
    state.sessions[event.sessionID] = Session(...)

    // 建立窗口 → session 映射
    if let tabId = s.termSessionID ?? s.tmuxPane ?? s.weztermPane {
        let key = TermTabKey(hostApp: s.hostApp.rawValue, tabId: tabId)
        state.sessionByTab[key] = event.sessionID
    }
    // VS Code 等无稳定 ID 的，不建映射，靠 Trigger 2
```

## 性能预算

- NSWorkspace 通知几乎零开销
- AppleScript 查 tab ID 有 ~50ms 延迟（每次窗口切换）
- 解析 jsonl 行 < 1 ms
- 状态更新都在主线程，UI 不会撕裂
